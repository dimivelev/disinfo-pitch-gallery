Disinfo.Analyzer

A hackable, self-hostable multi-agent disinformation pipeline — a FastAPI service orchestrating four CrewAI agents over fine-tuned transformers, streamed live into a Chrome MV3 extension.

CrewAI + Gemini 2.5 Flash FastAPI · SSE streaming Docker → Cloud Run

$ git clone github.com/<org>/disinfo-analyzer

TL;DR For Engineers

You POST a page's HTML. You get a live SSE stream of per-agent verdicts back.

🧩

Composable

Each analysis dimension is a CrewAI BaseTool wrapping one HF model. Add or swap tools without touching the rest.

🔒

Stateless

A fresh Crew is built per request — isolated state, safe concurrency, no cross-talk between analyses.

📡

Streaming

Server-Sent Events push status + result JSON as each agent finishes. No polling, no blocking waits.

Architecture

Four layers, one data path

1
Chrome MV3
activeTab · scripting · offscreen — grabs live tab HTML, calls the API
2
FastAPI
POST /analyze-html/ · Newspaper3k parse · builds a Crew per request
3
CrewAI Crew
4 agents run sequentially, reasoning via Gemini 2.5 Flash
4
HF Models
Each agent tool runs a fine-tuned transformer on PyTorch
extension↔ HTTP↔ FastAPI↔ CrewAI↔ Transformers↑ SSE back
Request Lifecycle

One request, a stream of events

POST /analyze-html/  Content-Type: application/json
{ "html": "<!doctype html>…" }

↓ text/event-stream
event: status  data: {"agent":"headline","state":"running"}
event: result  data: {"agent":"headline", …}
event: status  data: {"agent":"emotion","state":"running"}
event: result  data: {"agent":"emotion", …}
…bias, conspiracy…
event: done    data: {"result": { …final JSON… }}
  • Parse headline + body with Newspaper3k.
  • Instantiate a fresh, isolated Crew per request.
  • Agents run sequentially; each emits status then result.
  • Client renders incrementally — UI fills in as events arrive.

Why per-request crews? Isolated state means safe concurrency — no shared memory bleeding one page's analysis into the next.

The API

Two endpoints. That's the whole surface.

Contract

POST /analyze-html/
  req  { "html": string }
  res  200 text/event-stream
       (status → result ×4 → done)

GET  /            # health check
  res  200 { "status": "ok" }

Notes

  • CORS open — call it from any extension or origin.
  • Body is raw page HTML; server owns the parsing.
  • StreamingResponse keeps the connection alive per agent.
  • Final done event carries the consolidated JSON result.
Extending It

Add an agent = add a BaseTool + Task.

STEP 1

Wrap a model

Subclass BaseTool, load your HF pipeline in _run(), return structured output.

STEP 2

Wire an Agent + Task

Give the agent the tool, describe the Task, append it to the sequential Crew — it slots into the SSE stream automatically.

SWAP

Change models freely

Point a tool at a different HF checkpoint — the contract to the crew is just text in, JSON out.

Performance & Design Notes

The trade-offs we made on purpose

📦

Model loading

Transformers + PyTorch dominate cold start — the image is heavy, so keep instances warm on Cloud Run.

✂️

Truncation

Encoder models cap at their max sequence length; body text is truncated to fit before classification.

🌡️

temperature 0.1

Gemini 2.5 Flash runs near-deterministic — reasoning over model outputs stays stable and reproducible.
